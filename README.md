# 个性化智能旅游与校园导览系统设计方案

**技术栈：后端 (Python + FastAPI/Flask) | 前端 (Vue3 + Element Plus/高德 JS API)**

---

## 1. 旅游推荐

### 功能需求

- 用户可以根据自己的喜好选择不同的景点和学校作为旅游目的地。
- 在游览前，系统会向用户推荐旅游景点和学校，可以按照旅游热度、评价和个人兴趣进行推荐。
- **核心算法（Top-K排序）**：按照热度和评价排序；考虑到用户通常只看前10个，要求不经过完全排序排好前10，并考虑数据动态变化。
- **核心算法（查找与排序）**：输入名称、类别、关键字查询，多项结果按热度/评价排序。

### 前端实现功能 (Vue3)

- **独立页面**：旅游推荐作为系统首页或单独导航页。
- **UI布局**：包含“Top-10推荐景点卡片栏”和“顶部搜索栏”。推荐卡片需直观展示景点头图、综合热度值和星级评价。
- **搜索联想提示**：搜索栏绑定 `@input` 事件，输入文字时触发防抖请求，展示下拉关联提示框（类似谷歌搜索下拉词）。
- **动态视图切换**：执行搜索后，隐藏“推荐景点栏”，使用 `v-if` 切换至“搜索结果列表视图”。
- **排序与标识**：搜索结果页提供“按热度”、“按评价”排序的单选按钮；每个结果卡片上用特殊Icon（如“室内地图可用”）标识该景点是否存在内部详细信息。

### 后端实现功能 (Python)

- **数据存储设计**：静态信息（名称、坐标、是否有内部信息）存入 MySQL (`MySQL` 表)；动态数据（热度、评分）存入 Redis (`Hash` 或 `ZSet`)，利用 Redis 实现热度的实时递增更新。
- **倒排索引与查找API**：在内存或 Elasticsearch 中建立关键字/类别的倒排索引。开发搜索接口，接收关键字快速召回候选 ID 集合。
- **Top-10 不完全排序算法**：使用 Python 内置的 `heapq` 库，设计基于容量为 10 的**最小堆（Min-Heap）** 函数。遍历候选集时，从 Redis 实时读取热度/评分计算综合分数，将其推入堆中。若堆满且新元素分数大于堆顶，则 `heappop` 弹出堆顶并 `heappush` 插入新元素，实现 $O(N \log 10)$ 时间复杂度的前10名截取，完美避免全量排序。

## 2. 旅游路线规划

### 功能需求

- 单点/多点最短路径规划（途经多点后返回起点）。
- 图形界面导航（地图展示、路线高亮）。
- **多种策略**：最短距离、最短时间（结合自拟道路拥挤度计算实际速度）、交通工具时间最短（步行/自行车/电瓶车混合）。
- **室内导航**：模拟教学楼/博物馆的楼层结构，大门->电梯->楼层->房间导航。

### 前端实现功能 (Vue3)

- **地图基础展示**：引入 `@vuemap/vue-amap`（高德地图 Vue3 组件），加载目标地点的 2D/3D 地图。
- **起点交互**：用户点击地图取点，若未精确点在道路上，前端发送坐标给后端，后端返回“吸附”后的最近路网节点坐标并在地图上标记起点。
- **搜索与标点**：地图侧边栏提供场所搜索，结果以红标 (Marker) 显示在地图上。多个相似结果允许点击选中确认。
- **多点存储池**：底部或侧边提供一个“已选目的地列表 (Tag/List)”，用户点击红标后加入该列表，作为 TSP（途经多点）的输入。
- **路径可视化**：后端返回路径轨迹数组后，使用 `AMap.Polyline` 绘制路线。普通道路底图自带，最短路径用高亮蓝色（实线代表步行，虚线代表骑车）覆盖绘制。
- **室内地图切换**：当路径进入建筑物内部时，弹出覆盖层或切换地图实例，展示室内自定义平面图（Canvas/ImageOverlay），右侧出现“1F、2F、3F”楼层切换电梯按钮。

### 后端实现功能 (Python)

- **地图结构存储**：使用 `OSMnx` 爬取真实路网并转化为 `NetworkX` 图对象 (`Graph`)。节点存经纬度和类型，边存长度、拥挤度( $\le 1$ )、允许交通工具类型。持久化可采用 `PostGIS` 或序列化为 `.graphml` 文件。
- **图数据读取与转换函数**：开发 API 接收前端传来的起点经纬度和目标点 ID，利用 `scipy.spatial.KDTree` 实现经纬度到图节点的快速吸附（匹配阈值范围内的最近点）。
- **最短路径核心算法设计**：
    - 单点最短距离/时间：利用 `NetworkX` 的 `nx.dijkstra_path`，根据前端传参动态切换权重（`weight='length'` 或 `weight='time'`，其中 $time = length / (ideal\_speed \times congestion)$ ）。
    - **途经多点并返回（TSP核心）**：先用 Dijkstra 计算起点与所有选中目标点之间**两两的最短路径**（构建距离矩阵），然后使用 **状态压缩动态规划 (Bitmask DP)** 在 Python 中求解 TSP 最短回路，返回组合后的完整经纬度轨迹序列发给前端。

## 3. 旅游场所查询

### 功能需求

- 选中场所后，找附近一定范围内（如 500 米）超市、卫生间等，按**实际可达路径距离**排序（不能用直线距离）。
- 支持类别过滤和类别名称输入查询排序。

### 前端实现功能 (Vue3)

- **动态交互面板**：用户在地图上点击/选中某个景点后，弹出操作面板卡片。
- **双态按钮**：
    - 按钮 A：“进入景点内部”。通过判断返回数据中 `has_indoor` 字段，若为 `false` 则应用 CSS 的 `disabled` 样式置灰不可点；若为 `true` 则点击触发室内地图展示。
    - 按钮 B 组：“周边设施发现”（提供类似外卖App的横向滑动图标：公共卫生间 🚻、便利店 🏪、餐厅 🍴 等）。
- **搜索过滤与展示**：面板上方提供输入框支持自定义输入类别（如“咖啡馆”）。查询后，在侧边栏以列表形式展示结果，并直接标注出“实际步行距离：320米”。

### 后端实现功能 (Python)

- **周边设施数据源**：将所有场所（卫生间、超市）关联至最近的路网节点，打上 `category` 标签。
- **核心算法（路网 BFS/Dijkstra 等时圈搜索）**：
    - **切忌先查全部再算距离**。后端收到 `(起点ID, 类别, 最大搜索距离限度)` 后，以起点为源，启动 **有深度/距离限制的 Dijkstra 算法**。
    - 算法沿路网的边向外扩散，每当弹出的节点包含目标 `category`（如卫生间）时，记录该节点以及当前累加的路径距离。
    - 当搜索半径超过设定值（如实际路程 1000 米）或找到足够多的数量时，停止搜索。
- **核心算法（快速排序）**：将搜集到的目标设施集合，按刚才累加得出的“实际可达路径距离”使用 **快速排序 (Quick Sort)** 进行升序排列，并截取返回前端。

## 4. 旅游日记管理与交流

### 功能需求

- **管理**：图文/视频记录，统一管理，浏览量即热度，可评分。推荐需基于热度/评分进行排序。
- **交流**：按目的地查询并排序；精确名称查找（高效）；按内容**全文检索**；日记**无损压缩**存储。

### 前端实现功能 (Vue3)

- **日记发布器**：引入富文本编辑器（如 `WangEditor` 或 `Quill`），支持文本输入、拖拽上传图片/短视频，点击发布向后端发送 JSON 或 FormData 数据。
- **日记广场与推荐瀑布流**：首页展示系统推荐的游记，显示浏览量(热度)图标 `👁️ 1024` 和星级评分 `⭐ 4.8`。点击进入详情页后，触发后端的“热度+1”接口。
- **多维搜索栏**：提供一个复合搜索框，左侧放置下拉选择器（可选模式：目的地 / 标题精确查找 / 全文检索），用户输入文字后展示相应的结果列表。

### 后端实现功能 (Python)

- **精确查询算法 (B+树)**：将日记的标题 `title` 字段在 MySQL 数据库中建立普通/唯一索引（底层原理为 B+ 树）。在日记数量极大且频繁插入时，保证 $O(\log N)$ 级别的高效精确匹配查询。
- **全文检索算法**：集成 **Elasticsearch** (或在纯 Python 中实现轻量级 `Whoosh` 库)。对日记的正文内容进行倒排索引构建。查询时使用 TF-IDF / BM25 算法计算关键词与日记内容的相关性得分并返回结果。
- **热度与推荐排序算法**：浏览量实时存入 Redis `Hash`，基于（浏览量*权重 + 评分*权重）计算推荐分，复用第 1 部分的 **最小堆 (Min-Heap)** 算法选取 Top-N 推荐。
- **无损压缩存储算法**：
    - 在后端接收到前端传来的富文本内容/长字符串后，使用 Python 内置的 `zlib` 或 `gzip` 模块（底层核心算法为 **LZ77 字典压缩 + Huffman 霍夫曼编码**）。
    - 将庞大的文本无损压缩为二进制 `bytes` 后，存入对象存储（如 MinIO）或数据库的 `BLOB` 字段中；数据库仅保存文件路径或哈希值，极大地节约存储空间。读取时通过 `zlib.decompress` 解压返回给前端。

## 5. 美食推荐

### 功能需求

- 选中景点后，推荐附近美食（热度、评价、距离排序，菜系过滤）。
- 前 10 个美食不完全排序。
- 支持基于美食名称、菜系、窗口名称的模糊查询及多项结果综合排序。

### 前端实现功能 (Vue3)

- **景点挂载面板**：与第 3 部分类似，点击地图景点后，在 Tab 栏中提供“美食推荐”专属面板。
- **条件过滤栏**：包含“菜系选择”下拉框（川菜、粤菜、快餐等），以及三个排序切换标签（🔥按热度、⭐按评价、🚶按距离）。
- **模糊搜索框**：一个大搜索框，提示文字“输入想吃的美食、档口名或菜系...”。
- **Top-10 榜单展示**：展示带有美食图片的列表，列表最右侧清晰标出距离起点（当前选中景点）的实际步数或米数。

### 后端实现功能 (Python)

- **模糊查找算法**：
    - 方法一：使用 Elasticsearch 的 N-gram 分词器进行模糊匹配。
    - 方法二（纯算法实现）：构建 **字典树 (Trie)** 用于前缀匹配，结合 **Levenshtein 编辑距离 (Edit Distance) 动态规划算法** 计算用户输入字符串与饭店名/窗口名的相似度，容忍错别字（如将“麻辣烫”错输为“麻辣糖”，编辑距离为1，依然可以召回）。
- **距离计算**：利用第 2、3 部分构建的路网图，快速计算当前景点坐标节点到各饭店坐标节点的实际路网最短距离。
- **综合排序与 Top-10 (最小堆)**：
    - 设计评分公式：$TotalScore = \alpha \times Heat + \beta \times Rating - \gamma \times Distance$（距离越近，分数越高）。
    - 对模糊查询或类别过滤召回的候选美食集，遍历计算上述综合分数。
    - 使用容量为 10 的 **最小堆 (Min-Heap)** 维护当前得分最高的前 10 个美食。由于只需前 10 名，遍历一遍 $O(N \log 10)$即可完成筛选，满足不进行完全排序的要求，并作为 JSON 返回前端渲染。
